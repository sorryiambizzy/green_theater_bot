# Publish to GitHub + Autodeploy + CLAUDE.md

## Goal

Three related outcomes:
1. A `CLAUDE.md` so a future Claude Code session understands this project without the user re-explaining context.
2. The version running on the VDS always matches the version in the repo, with minimal manual effort.
3. The project published as a public pet-project on GitHub, with setup docs, so others (and future-you) can read/run it.

## Context

- Local repo: `/Users/epodkin/Вижуалка/rutracker-bot`, git history is clean (no real credentials ever committed — verified via `git log --all -p` scan).
- `.env` holds real secrets (`RUTRACKER_USERNAME`, `RUTRACKER_PASSWORD`, `TELEGRAM_BOT_TOKEN`) and is gitignored; `.env.example` has placeholders and is tracked.
- Deployed copy lives on a VDS (<VDS_IP>) at `/opt/rutracker-bot`, run via `rutracker-bot.service` (systemd, `User=root`). It is currently just files copied via `scp`, not a git checkout.
- VDS specs: 1 vCPU, ~1GB RAM (+2GB swap file added earlier this session), no other services of note.
- Repo name: `green_theater_bot`, public, MIT license.

## Decisions

### 1. CLAUDE.md

Lives at repo root, conceptual/architectural — not a rehash of file contents:
- What the bot does (Telegram → rutracker.org search/download bridge)
- Why login uses Playwright (rutracker requires a real browser session + captcha solving relayed through Telegram; plain `requests`-based login isn't sufficient)
- The threading model in `playwright_login` (sync Playwright driven from a background thread, coordinated with the asyncio event loop via two queues) — this is the least obvious part of the codebase and worth explaining so a future session doesn't have to re-derive it
- Known fragility: `_parse_search_results` depends on rutracker's current HTML markup (class names) and silently returns zero results if the site changes markup, rather than erroring — worth flagging as a maintenance trap
- VDS constraints (1 vCPU, low RAM) and why that matters (headless Chromium is heavy relative to the box)
- Pointer to the deploy process (§2) instead of duplicating it

No secrets, no server IP/credentials — this file ships in the public repo.

### 2. GitHub repository

- `gh repo create green_theater_bot --public --license mit --source=. --remote=origin`
- Add `LICENSE` (MIT, generated by `gh repo create`)
- Add `README.md`: what the bot does, local setup (`venv`, `pip install -r requirements.txt`, `playwright install chromium`), `.env` variables (referencing `.env.example`), how to run locally, how the systemd deploy works, VDS sizing note (2 vCPU recommended, 1 vCPU works with swap but logins are slower)
- Push existing commit history as-is (already scanned clean)

### 3. Autodeploy on push to `main`

- `/opt/rutracker-bot` on the VDS becomes an actual `git clone` of the new GitHub repo (currently it's just loose files) — reconciled by cloning fresh into a temp dir, copying the live `.env`, and swapping it in, so the running secrets survive the switch.
- `.github/workflows/deploy.yml`: triggers on push to `main`, SSHes into the VDS (`appleboy/ssh-action` or plain `ssh` with `sshpass`), runs:
  ```
  cd /opt/rutracker-bot
  git fetch origin main
  git reset --hard origin/main
  pip install -r requirements.txt        # only meaningfully slow if deps changed
  systemctl restart rutracker-bot
  ```
- GitHub Secrets: `DEPLOY_HOST` (<VDS_IP>), `DEPLOY_USER` (root), `DEPLOY_PASSWORD` (existing root password — user's explicit choice over provisioning a dedicated deploy key, accepted for this pet project).
- `git reset --hard` is safe here because `/opt/rutracker-bot` is a deploy target, not a place where anyone edits code directly — no local commits to lose.
- This deploy step itself is lightweight (network fetch + restart, no build/compile), so it doesn't stress the 1-vCPU box — the earlier resource concern was specifically about headless Chromium during login, not about deploys.

## Out of scope

- Not changing the login/captcha flow, parsing logic, or systemd unit itself — those are already fixed and working as of this session.
- Not switching the VDS off root/away from password auth for general SSH use — only the deploy secret is being formalized as a GitHub secret using the existing credential.
- Not adding CI test runs (pytest) before deploy in this pass — could be a follow-up but wasn't requested.

## Testing / Verification

- After pushing: confirm the GitHub Actions run succeeds, confirm `systemctl status rutracker-bot` shows a fresh process start time on the VDS, confirm `/opt/rutracker-bot` is a real git repo (`git status` clean, `git log` matches GitHub).
- Confirm the bot still responds in Telegram after an autodeploy (smoke test, not full regression).
