# Публикация на GitHub + автодеплой + CLAUDE.md

## Цель

Три связанные задачи:
1. `CLAUDE.md`, чтобы будущая сессия Claude Code понимала проект без повторного объяснения контекста.
2. Версия на VDS всегда совпадает с версией в репозитории, с минимумом ручных действий.
3. Проект опубликован на GitHub как публичный pet-проект, с документацией по установке — чтобы его могли прочитать и запустить другие (и будущий ты).

## Контекст

- Локальный репозиторий: `/Users/epodkin/Вижуалка/rutracker-bot`, история git чистая (реальные креды туда никогда не попадали — проверено через `git log --all -p`).
- `.env` содержит реальные секреты (`RUTRACKER_USERNAME`, `RUTRACKER_PASSWORD`, `TELEGRAM_BOT_TOKEN`) и в git не трекается; `.env.example` с плейсхолдерами — трекается.
- Развёрнутая копия живёт на VDS (<VDS_IP>) в `/opt/rutracker-bot`, запускается через `rutracker-bot.service` (systemd, `User=root`). Сейчас это просто файлы, скопированные через `scp`, а не git-репозиторий.
- Характеристики VDS: 1 vCPU, ~1GB RAM (+2GB swap-файл, добавленный ранее в этой сессии), других значимых сервисов нет.
- Имя репозитория: `green_theater_bot`, публичный, лицензия MIT.

## Решения

### 1. CLAUDE.md

Лежит в корне репозитория, содержит концептуальную/архитектурную информацию — не пересказ содержимого файлов:
- Что делает бот (мост между Telegram и поиском/скачиванием на rutracker.org)
- Почему логин реализован через Playwright (rutracker требует настоящую браузерную сессию + решение капчи, транслируемой через Telegram; простого логина через `requests` недостаточно)
- Модель работы с потоками в `playwright_login` (синхронный Playwright, управляемый из фонового потока, синхронизированный с asyncio event loop через две очереди) — это самая неочевидная часть кодовой базы, стоит объяснить, чтобы будущей сессии не пришлось заново её распутывать
- Известная хрупкость: `_parse_search_results` зависит от текущей HTML-разметки rutracker (имена классов) и молча возвращает ноль результатов, если сайт меняет разметку, вместо явной ошибки — стоит отметить как ловушку для будущей поддержки
- Ограничения VDS (1 vCPU, мало RAM) и почему это важно (headless Chromium тяжёл относительно возможностей сервера)
- Ссылка на процесс деплоя (§2) вместо дублирования

Никаких секретов, IP сервера или кредов — этот файл попадёт в публичный репозиторий.

### 2. Репозиторий на GitHub

- `gh repo create green_theater_bot --public --license mit --source=. --remote=origin`
- Добавить `LICENSE` (MIT, генерируется через `gh repo create`)
- Добавить `README.md`: что делает бот, локальная установка (`venv`, `pip install -r requirements.txt`, `playwright install chromium`), переменные `.env` (со ссылкой на `.env.example`), как запустить локально, как устроен деплой через systemd, рекомендации по размеру VDS (2 vCPU предпочтительнее, 1 vCPU работает со swap, но логин медленнее)
- Запушить существующую историю коммитов как есть (уже проверена и чиста)

### 3. Автодеплой при push в `main`

- `/opt/rutracker-bot` на VDS становится настоящим `git clone` нового GitHub-репозитория (сейчас это просто файлы) — переход делается через клонирование во временную папку, перенос текущего живого `.env` и подмену директории, чтобы рабочие секреты пережили переключение.
- `.github/workflows/deploy.yml`: срабатывает при push в `main`, заходит по SSH на VDS (`appleboy/ssh-action` либо обычный `ssh` через `sshpass`), выполняет:
  ```
  cd /opt/rutracker-bot
  git fetch origin main
  git reset --hard origin/main
  pip install -r requirements.txt        # заметно долго только если зависимости изменились
  systemctl restart rutracker-bot
  ```
- Секреты в GitHub: `DEPLOY_HOST` (<VDS_IP>), `DEPLOY_USER` (root), `DEPLOY_PASSWORD` (существующий root-пароль — осознанный выбор пользователя вместо отдельного deploy-ключа, принят для pet-проекта).
- `git reset --hard` здесь безопасен, поскольку `/opt/rutracker-bot` — это цель деплоя, а не место, где кто-то правит код руками напрямую — терять локальные коммиты там нечего.
- Сам шаг деплоя лёгкий (сетевой fetch + рестарт, без сборки/компиляции), поэтому не нагружает слабый VDS — предыдущее опасение по поводу ресурсов касалось конкретно headless Chromium при логине, а не деплоя.

## Вне рамок задачи

- Не меняем логику логина/капчи, парсинг или сам systemd-юнит — они уже починены и работают по итогам этой сессии.
- Не переводим VDS с root/парольного SSH на что-то другое для общего доступа — формализуется через GitHub Secret только секрет для деплоя, с использованием уже существующего пароля.
- Не добавляем прогон тестов (pytest) перед деплоем в этом заходе — может быть отдельным шагом позже, но явно не запрашивалось.

## Тестирование / проверка

- После push: убедиться, что прогон GitHub Actions завершился успешно, что `systemctl status rutracker-bot` показывает свежее время старта процесса на VDS, что `/opt/rutracker-bot` — настоящий git-репозиторий (`git status` чист, `git log` совпадает с GitHub).
- Убедиться, что бот по-прежнему отвечает в Telegram после автодеплоя (smoke-тест, не полная регрессия).
